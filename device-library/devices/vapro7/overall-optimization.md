# VAPro7 / WellnessHub 联动优化方案

## 1. 统一原则

- WellnessHub 后台负责配置。
- VAPro7 设备端负责测量与结果分发。
- 手机端 H5 负责查看完整报告。
- 同一项目的配置、设备显示和报告显示必须保持一致。

## 2. 配置关系

### 后台负责

- 测量项目启用 / 停用
- 项目下报告显隐
- 门店默认配置
- 指定设备覆盖

### 设备端负责

- 按后台配置展示入口
- 按后台配置显示结果与报告入口
- 保留测量流程、异常流程和重测流程

### H5 负责

- 展示手机端完整报告
- 与设备端、后台显示同一份结果内容

## 3. 文档组织建议

- **同一份 PRD**：适合描述同一业务链路上的跨端能力。
- **分文件**：只在某个端有独立流程、独立权限或独立交互时再拆。
- 当前这类需求建议：
  - 一份主 PRD
  - 按端拆分章节
  - 不要按端拆成多份互相割裂的文档
  - 如果 Windows App 只是另一种客户端展示，同样放进这份主 PRD 的「客户端差异」章节即可；只有它有独立登录、权限、导航或硬件流程时才单独拆分。

### 3.1 本次推荐结论

- **WellnessHub 后台 + VAPro7 设备端 + 手机 H5**：放在同一份主 PRD 里描述，因它们是同一条测量-报告链路。
- **Windows App**：
  - 如果只是同样消费这条链路、只是另一个展示端，仍放在同一份主 PRD 的客户端差异章节；
  - 如果它有独立的登录、权限、导航、设备控制或发布节奏，再单独拆文件。
- 当前阶段优先级：
  1. 先统一一份主 PRD
  2. 再按端拆实现章节
  3. 最后才考虑拆出 Windows App 独立文档

## 4. VAPro7 设备端 demo 优化方向

- 删除设备端里的项目配置入口。
- 保留测量入口、测量中引导、结果页、报告页。
- 新增异常流程：
  - 姿势不正确
  - 测量后直接原地重测
  - 手势不可用时按键兜底
- 首页只展示后台已启用的项目，不负责编辑配置。

## 5. WellnessHub demo 优化方向

- 门店信息管理中新增“测量项目与报告配置”。
- 配置内容只保留：
  - 项目开关
  - 项目下报告显隐
  - 门店默认 / 指定设备覆盖
- 不再扩展成多页面 demo。
- 页面只保留编辑态和设备端效果预览。

## 6. 推荐结论

- 如果 Windows App 只是另一个配置展示端，先和后台放在同一份 PRD 的独立章节里。
- 如果 Windows App 有独立入口、权限、发布或交互，再单独拆 PRD。
- 当前优先级：先统一主 PRD，再拆分实现页。

## 7. 当前 Demo 审查结论

- 已完成的部分：
  - WellnessHub 后台已收敛为单页编辑态 demo
  - 测量项目配置已迁移到后台，不再放在设备端编辑
  - 设备端已补充原地重测、姿态异常、手势兜底等异常流程
  - 后台 / 设备端 / H5 的关系已经在主 PRD 中统一
- 仍需补齐的部分：
  - 设备端 demo 还缺一次浏览器级视觉 QA
- 当前仍保留的评审 / 预览性质页面：
  - `standard-next-step-touch.html`
  - `single-flow-placeholder.html`
  - `结果方案三.html`
  - 首页底部运营位示意
- 说明：
  - 这些页面可以保留为评审入口或兼容入口，但不能作为“已完全去 demo 化”的证据
- go-live 判断建议：
  1. 先做一轮页面级验收，检查文案、交互、异常态和空态
  2. 再确认设备端 / 后台 / H5 的链路一致性
  3. 最后再判断是否可直接交付给用户使用

## 8. 最终优化收口

- `WellnessHub 后台 + VAPro7 设备端 + 手机 H5`：建议继续放在同一份主 PRD 里，避免配置、入口、报告三端脱节。
- `Windows App`：默认不单独拆文件；只有当它有独立的登录、权限、导航、硬件流程或发布节奏时，才单拆成独立文档。
- 当前设备端 demo 的关键业务能力已经补齐，但仍不能直接宣称“可直接上线给用户使用”，原因只剩浏览器级视觉 QA 和最终文案复核。
- 当前设备端 demo 的主入口已收口，主页与设备设置页不再暴露“评审入口 / 运营位示意”；剩余的 `standard-next-step-touch.html`、`single-flow-placeholder.html`、`结果方案三.html` 仅作为隔离的预览/兼容页存在，不进入主流程。
- 当前 demo 中的评审 / 预览页面不应与正式链路混为一谈；正式链路应只保留已收口的测量准备、测量中、完成页、结果页与异常兜底。
- 现阶段最优先的收口动作：
  1. 做全链路浏览器级验收
  2. 核对首页、结果页、异常态文案是否无歧义
  3. 再决定是否进入正式交付
