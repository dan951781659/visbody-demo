# WellnessHub 用户数据看板 PRD（V1）

## 1. 项目背景与目标

### 1.1 背景

WellnessHub 已具备：测量报告、用户分配、用户管理（健康档案/训练大纲状态）、AI 报告解读、AI 对话、多门店等能力。  
当前缺少可供门店管理者/教练直接决策的聚合看板，需新增「用户数据看板」作为运营第一入口。

### 1.2 目标

- 管理者：快速判断资产规模、运营效率、风险用户、当日行动优先级。
- 教练：在本人权限范围内掌握会员活跃/复测/分层，支持导出与下钻。

### 1.3 非目标（V1）

- 不做 CRM 工单流转（跟进、标记已处理、提醒消息下版本由 AI 助手承接）。
- 不做外部系统（收银/CRM）打通。

---

## 2. 与销售 V1 需求对齐结论

### 2.1 可做项（V1）

- P1 资产总览：累计测评用户、本月新增、活跃、复测、沉睡。
- P2 画像结构：性别、年龄、目标分布。
- P3 漏斗（核心版）：已测评 -> 已生成训练计划（大纲） -> 已复测。
- P4 分层：高潜、待复测、流失风险、持续改善（阈值门店可配置）。
- P5 名单：统计、下钻、导出。

### 2.2 口径说明（已确认）

- 活跃用户（V1）：近 30 天满足任一：
  - 有非导入测评；
  - 训练大纲为执行中；
  - 期内生成/更新训练大纲；
  - 期内 AI 报告解读新生成；
  - 期内 AI 对话有记录。
- 沉睡用户：同周期内不满足活跃定义。
- 导入历史数据：不计入测量次数与已测评漏斗节点。
- 已查看报告：非 V1 核心节点。

## 3. 角色与权限

### 3.1 角色定义

- 总门店管理员（Pro 总门店）
- 子门店管理员（单店或 Pro 子店）
- 教练（分配自己）

### 3.2 可见范围


| 角色     | 数据范围               | 默认可见模块                           | 不展示                     |
| ------ | ------------------ | -------------------------------- | ----------------------- |
| 总门店管理员 | 全部/指定子门店全量数据       | P1-P5 + 运营补充 + Pro 模块 + 导出       | 无                       |
| 子门店管理员 | 当前门店全量数据           | P1-P5 + 运营补充 + Pro 模块（若 Pro）+ 导出 | 子门店对比（仅总店）              |
| 教练     | 本人归属用户（含该用户历史全部报告） | P1/P3/P4/P5（本人池）+ 趋势 + 导出        | 待分配、跨教练排行、跨门店对比、全店设备利用率 |


### 3.3 特殊规则（已确认）

- 普通教练不展示「待分配用户数」。
- 用户归属教练后，该教练统计可见该用户历史全部报告（按当前分配归属，不按快照归属）。

## 5. 指标体系

### 5.1 指标合并原则（去重后）

- 统一“一个指标只保留一个主定义”，不再在不同页面重复定义同名指标。
- 按角色分为三层：**通用核心指标**（总部/馆主都看）、**总部专属指标**、**馆主专属指标**。
- 指标优先级：先保留可直接驱动动作的指标（可下钻、可导出），再保留解释性指标。
- 口径冲突时，以“可被测试复算”为准，避免纯展示指标。
- 合并策略为“融合增强”：保留原方案中已验证有效的运营指标，不做全量替换。

### 5.2 通用核心指标（总部端 + 馆主端）

- 用户规模与活跃：累计测评用户、本期新增测评、活跃用户、沉睡用户。
- 闭环效率：复测用户、复测达成率、服务转化进度（已测评 -> 已生成计划 -> 已复测）。
- 用户结构：性别分布、年龄分布、目标分布、问题标签分布。
- 重点人群：高潜转化、待复测、流失风险、持续改善。
- 行动执行：今日重点行动名单（支持筛选、排序、下钻、导出）。

### 5.3 总部端（总门店管理员）专属指标

- 门店达标率（达成总部目标门店占比）。
- 高风险门店数（经营健康评分超阈值）。
- AI 渗透门店占比（有 AI 解读或 AI 对话行为）。
- 子门店横向对比（测评、活跃、复测率、风险人数）。
- 设备健康率（总部维度）。

### 5.4 馆主端（子门店管理员）专属指标

- 待分配用户数（服务承接缺口）。
- 计划执行率（执行中大纲占比）。
- 门店复测达成率（复测用户 / 已测评用户）。
- 高风险会员数（门店内流失风险池）。
- 待绑定报告数（前台补绑效率）。

### 5.5 交互入口与页面切换

- 顶栏增加“视图切换入口”：**总部端（PC） / 馆主端（PC） / 教练端（PC）**。
- 视图与角色一一对应：
  - 总部端（PC）= 总门店管理员
  - 馆主端（PC）= 子门店管理员
  - 教练端（PC）= 教练
- 切换视图后，指标卡、可见模块、下钻范围实时切换并保持统一筛选条件（门店、时间、角色）。

### 5.6 原方案保留指标（明确保留，不替换）

- 活跃用户（V1 口径）、沉睡用户（活跃补集）。
- 复测用户、待复测用户、流失风险用户、持续改善用户。
- 待分配用户（管理者可见）、今日重点行动名单。
- AI 解读覆盖率、AI 对话记录用户占比、执行中大纲占比。
- 运营补充：待绑定报告、未分配占比、设备利用率、健康档案完整度。

## 6. 数据统计规则

### 6.1 时间规则

- 默认范围：近 7 天 / 30 天 / 90 天 / 累计 / 自定义
- 统计按门店时区（日边界 00:00-24:00）

### 6.2 去重与归属

- 用户去重键：平台用户唯一 ID
- 教练归属：按当前用户-教练关系，且包含该用户历史全部报告

### 6.3 Basic / Pro

- Basic 不展示或置灰 Pro 指标（AI 解读、训练大纲深度指标）
- Pro 展示完整 AI 模块

### 6.4 Dashboard 历史导入

- 导入历史数据不计入测量次数、已测评漏斗节点及相关趋势
- 可选新增「导入档案量」独立指标

### 6.6 研发落地约束（必须满足）

- 统一过滤器：所有指标必须受同一组筛选条件控制（角色、门店、时间范围、导入排除规则）。
- 指标可解释：每个指标接口必须返回 `source_tables`、`formula_desc`、`denominator_desc` 元数据，供前端展示“来源与依据”。
- 指标可下钻：所有可点击指标必须返回对应明细查询参数，禁止前端“猜参数”。
- 口径版本化：规则变更需写入 `metric_version`（如 `active_user_v1`），用于历史对比与回溯。
- 安全与脱敏：导出手机号默认脱敏（中间4位），仅具备导出敏感信息权限时可明文。

## 7. 页面结构（信息架构落地）

1. 顶栏：角色视图切换（总部端/馆主端/教练端）+ 页面切换（总览页/分析页）+ 门店筛选 + 时间筛选 + 刷新 + 导出
2. 总览页：P1 资产总览 KPI + 趋势效率 + 转化进度 + 今日行动名单
3. 分析页：用户画像 + 分层机会 + 运营补充 + 角色专属分析卡
4. 总部端分析重点：门店达标率、高风险门店、AI 渗透门店、子门店对比、设备健康
5. 馆主端分析重点：待分配用户、计划执行率、复测达成率、高风险会员、待绑定报告
6. 底部：Pro AI 模块或 Basic 升级引导

### 7.1 UI 视觉规范（正式版）

- 信息密度参考飞书数据看板：卡片化、紧凑间距、主数值突出、弱化装饰元素。
- 主色调沿用 `training-outline` 页面色板：
  - 主色 `#165DFF`
  - 主色浅底 `rgba(22,93,255,0.08)`
  - 背景底色 `#F2F6FF`
  - 辅助文本 `#62708A`
- P1 KPI 卡片必须支持：指标标题、主值、环比标签（上升/下降）、口径说明，且整卡可点击下钻。
- 「运营补充」采用小卡片概览，不再使用标签堆叠；每个卡片可直接进入对应明细。

### 7.2 二级下钻页字段规范（按指标）

> 原则：下钻字段必须与指标语义强关联，禁止通用表头复用所有指标。


| 指标类型                             | 下钻表格字段（至少）                                           | 说明               |
| -------------------------------- | ---------------------------------------------------- | ---------------- |
| 用户类指标（累计测评、新增、活跃、复测、沉睡、待分配、分层命中） | 用户ID、姓名、性别、年龄、身高(cm)、体重(kg)、手机号、门店、归属教练、最近行为         | 作为“运营动作名单”标准字段   |
| 用户画像（性别/年龄/目标/问题标签）              | 用户ID、姓名、性别、年龄、身高(cm)、体重(kg)、手机号、门店、归属教练、命中标签/目标、最近行为 | 必须可支持导出用于分群触达    |
| AI 解读覆盖率                         | 报告ID、用户、门店、解读状态、解读时间、报告来源设备                          | 体现“报告维度”而非用户通用维度 |
| AI 对话记录用户占比                      | 用户ID、姓名、门店、对话轮次、最近对话时间、归属教练                          | 体现会话活跃度          |
| 执行中大纲占比                          | 用户ID、姓名、门店、大纲状态、最近更新时间、归属教练                          | 体现计划执行状态         |
| 待绑定报告数                           | 报告ID、上传时间、来源设备、门店、候选用户数、处理状态                         | 用于前台补绑定          |
| 设备利用率 / 设备贡献                     | 设备ID、设备型号、门店、在线天数、测评次数、利用率                           | 设备运营视角           |
| 子门店对比                            | 门店、累计测评用户、活跃用户、复测率、流失风险用户、设备利用率                      | 总店横向对比           |


### 7.3 下钻页「来源与依据」展示规范

- 固定展示三块内容：**数据来源**、**计算逻辑**、**分子/分母公式**。
- 不再使用含义不明确的小趋势占位图；改为文本化公式卡，示例：
  - 分子：已解读报告数
  - 分母：可解读报告数（剔除导入历史）
  - 结果：AI 解读覆盖率

### 7.4 客户可讲解文案规范（演示/售前）

- 页面标题和模块名优先使用业务语言，避免技术缩写（示例：`P3 漏斗` -> `服务转化进度`）。
- 提示语要包含“看什么 + 可以做什么”（示例：`点击某一天查看明细名单`）。
- 操作文案使用明确动词（示例：`导出数据（汇总+名单）`、`配置判定阈值`）。
- 说明区采用门店负责人能直接复述的表达，不使用内部术语堆叠。
- PRD 与页面名词必须一致，避免同一概念多种叫法。

### 7.5 交互风格决策（总部端/馆主端）

- 采用老板 Demo 的交互思路：先选视图角色，再看对应指标与动作列表。
- 采用理由：
  - 更符合真实管理路径（总部先看门店经营、馆主先看门店执行）。
  - 降低认知切换成本（同一入口切角色，不跳系统）。
  - 有利于培训与售前演示（讲解路径清晰、模块对应明确）。
- 风险控制：若外部 Demo 存在鉴权不可访问，正式版以本 PRD 的可测试规则为准，不依赖外部页面实现细节。

### 7.6 UI 与交互对齐项（本轮实现）

- 左侧导航采用老板 Demo 的 PC 结构：品牌区 + 一级功能入口 + 当前激活态。
- 顶部采用“角色切换 + 页面切换 + 周期切换”组合交互，降低页面跳转成本。
- 分析页采用“主分析区 + 侧边 AI 助手”结构（馆主端支持展开/收起）。
- 保持原有下钻与导出链路不变，避免因样式迭代影响数据使用流程。

## 8. 测试方案与验收标准（V1）

### 8.1 功能验收（产品视角）

- 登录后默认首屏为数据看板（有权限账号）。
- 角色可见范围与权限矩阵完全一致（含教练隐藏项）。
- 导入历史数据不混入测量次数与“已测评”节点。
- P4 阈值可配置，保存后次日统计生效。
- P5 支持查看/筛选/下钻/导出，不含工单流转。
- 指标下钻字段与指标语义强关联（见 7.2）。

### 8.3 非功能测试要求

- 性能：默认筛选条件下，首屏接口 P95 < 2s；切换筛选 P95 < 3s。
- 一致性：同一筛选条件下，看板数值与下钻明细聚合值一致（允许缓存延迟范围内误差 0）。
- 可用性：所有可点击指标均有 hover 态与可点击提示；空数据状态有解释文案。
- 安全性：导出敏感字段遵循权限控制与脱敏策略；操作日志可审计。
