# 移动端「创建桌面快捷方式」- 产品需求文档

> **编制说明**：本文档定义在手机、平板等移动端浏览器中，引导用户将网站添加至主屏幕（桌面快捷方式）的产品能力，包含交互流程与各浏览器下的操作指引。  
> **关联交付**：`桌面快捷方式-移动端演示.html`、`manifest.json`、`sw.js`

---

## 一、需求背景

### 1.1 业务背景

- 产品以 H5/Web 形式在移动端使用，用户通过浏览器访问
- 希望用户将站点「添加到主屏幕」，形成桌面图标，提升回访率与使用心智
- 不同操作系统、不同浏览器对「添加到主屏幕」的支持方式不一致，需统一引导与提示

### 1.2 目标用户与场景

| 维度 | 说明 |
|------|------|
| **目标用户** | 使用手机或平板浏览器访问站点的 C 端用户 |
| **使用场景** | 首次或再次通过浏览器打开站点时，在合适时机询问是否创建桌面快捷方式 |
| **设备范围** | Android 手机/平板、iOS（iPhone/iPad）、其他移动端浏览器 |

### 1.3 设计目标

1. **主动询问**：在移动端打开时，友好询问用户是否同意创建桌面快捷方式
2. **能触发则触发**：在支持系统级安装/添加的浏览器（如 Android Chrome）中，用户同意后优先调起系统对话框
3. **有兜底指引**：在不支持程序调起的浏览器（如 iOS Safari）中，提供清晰的按浏览器区分的操作步骤
4. **体验可控**：不打扰已添加过的用户、不重复骚扰已拒绝用户（可记录「已询问」状态）

### 1.4 现状与约束

- **Android**：部分浏览器支持 PWA 安装或「添加到主屏幕」，可通过 `beforeinstallprompt` 等能力调起系统弹窗
- **iOS / iPadOS**：系统不允许网页主动调起「添加到主屏幕」，仅能提供图文步骤引导用户手动操作
- **不同浏览器**：入口名称与位置差异大（如「添加到主屏幕」「安装应用」「加入主画面」等），需按浏览器给出对应提示

## 二、功能概述

### 2.1 核心能力

| 能力 | 说明 |
|------|------|
| **设备与环境识别** | 识别 Android / iOS、手机 / 平板、是否已从主屏幕打开（standalone），用于决定是否弹窗及展示哪类引导 |
| **征询弹窗** | 在移动端首次（或未记录拒绝时）展示底部抽屉/弹窗，询问「是否同意创建桌面快捷方式」，提供「同意」「暂不」 |
| **程序调起** | 在 Android 支持 PWA 的浏览器中，用户点击「同意」后调起系统「安装/添加到主屏幕」对话框 |
| **分浏览器引导** | 无法程序调起时，根据当前浏览器类型展示对应的图文步骤（见第四节） |
| **状态记忆** | 用户选择「暂不」或关闭弹窗后，可记录「已询问」状态，避免短期内重复打扰（如使用 localStorage） |

### 2.2 功能入口与触发时机

| 项目 | 说明 |
|------|------|
| **载体** | 移动端 H5 站点（需 HTTPS，Android 安装能力依赖 PWA 条件） |
| **触发时机** | 页面加载后延迟约 0.5–1 秒弹出，避免首屏即被遮挡 |
| **不触发条件** | 非移动端、已从主屏幕打开、本次会话或历史已选择「暂不」 |

### 2.3 弹窗与引导内容

| 元素 | 说明 |
|------|------|
| **标题** | 如「创建桌面快捷方式」 |
| **说明文案** | 如「将本页添加到主屏幕，下次可从桌面图标直接打开」 |
| **主操作** | 「同意」：触发安装或展示当前浏览器引导 |
| **次操作** | 「暂不」：关闭弹窗并记录已询问 |
| **引导面板** | 根据浏览器类型展示对应步骤（见第四节），可带示意图或简短步骤列表 |

## 三、用户场景与逻辑

### 3.1 场景一：Android 手机/平板（支持 PWA 调起）

1. 用户使用 Chrome/Edge 等支持 PWA 的浏览器打开站点
2. 满足 PWA 安装条件时，页面可捕获 `beforeinstallprompt`
3. 用户看到征询弹窗并点击「同意」
4. **系统弹出「添加到主屏幕」/「安装应用」对话框**，用户确认后桌面出现图标
5. 若未触发系统弹窗（如不满足安装条件），则展示「Android 通用或当前浏览器」的图文引导

### 3.2 场景二：iOS / iPad（Safari 或其它浏览器）

1. 用户使用 Safari 或其它浏览器打开站点
2. 系统不允许网页主动调起「添加到主屏幕」
3. 用户看到征询弹窗并点击「同意」
4. **展示当前浏览器的添加步骤**（如 Safari：分享 → 添加到主屏幕 → 添加）
5. 用户按步骤在浏览器内手动完成添加

### 3.3 场景三：已从主屏幕打开

1. 用户此前已添加过快捷方式，本次从桌面图标进入
2. 页面以 standalone 或类似方式运行
3. **不展示征询弹窗**，避免对已添加用户重复打扰

### 3.4 场景四：用户选择「暂不」

1. 用户点击「暂不」或点击遮罩关闭弹窗
2. 将「已询问」写入本地（如 localStorage）
3. 本次会话或后续访问在一定策略下不再自动弹出（可按产品策略设置是否过期）

## 四、不同浏览器的创建方式提示

> 以下为各常见浏览器下「添加到主屏幕」的操作步骤，用于在无法程序调起时展示给用户。产品与前端需按**当前检测到的浏览器/设备**展示对应一条，避免混用。

### 4.1 Android 手机 / 平板

#### 4.1.1 Chrome（安卓）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 点击浏览器右上角 **「⋮」**（更多） |
| 2 | 在菜单中选择 **「添加到主屏幕」** 或 **「安装应用」** |
| 3 | 在弹窗中确认名称后点击 **「添加」** 或 **「安装」** |

**备选表述**：菜单中也可能显示为「安装应用」或「将 [网站名] 添加到主屏幕」。

#### 4.1.2 Samsung Internet（三星浏览器）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 点击浏览器底部导航栏 **「菜单」**（三横线） |
| 2 | 选择 **「添加到主屏幕」** 或 **「添加页面到」→「主屏幕」** |
| 3 | 确认名称后点击 **「添加」** |

#### 4.1.3 Firefox（安卓）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 点击右上角 **「⋮」** 菜单 |
| 2 | 选择 **「安装」** 或 **「添加到主屏幕」** |
| 3 | 按提示确认添加 |

#### 4.1.4 Edge（安卓）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 点击底部或顶部的 **「⋯」** 菜单 |
| 2 | 选择 **「添加到主屏幕」** 或 **「将此应用安装到设备」** |
| 3 | 确认后点击 **「添加」** |

#### 4.1.5 其它 Android 浏览器（通用兜底）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 打开浏览器 **菜单**（多为右上角 ⋮ 或 ⋯） |
| 2 | 在菜单中查找 **「添加到主屏幕」**、**「安装应用」** 或 **「添加书签/快捷方式」** 等类似选项 |
| 3 | 按提示完成添加 |

---

### 4.2 iOS / iPhone

#### 4.2.1 Safari（iPhone）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 点击屏幕底部 **「分享」** 按钮（方框带向上箭头） |
| 2 | 在分享菜单中向下滑动，找到并点击 **「添加到主屏幕」** |
| 3 | 可修改名称，然后点击右上角 **「添加」** |

#### 4.2.2 Chrome（iPhone，内核仍为 Safari）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 点击底部 **「分享」** 图标（或右上角 **「⋯」** → 分享） |
| 2 | 在分享页中选择 **「添加到主屏幕」** |
| 3 | 点击 **「添加」** 完成 |

> 说明：iOS 上 Chrome 等第三方浏览器仍使用系统 WebView，添加方式与 Safari 一致，均为「分享 → 添加到主屏幕」。

#### 4.2.3 Edge / Firefox（iPhone）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 点击 **「分享」** 或菜单中的分享选项 |
| 2 | 在系统分享列表中选择 **「添加到主屏幕」** |
| 3 | 点击 **「添加」** 完成 |

---

### 4.3 iPad / iPadOS

#### 4.3.1 Safari（iPad）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 点击地址栏左侧或顶部的 **「分享」** 按钮（方框带向上箭头） |
| 2 | 在分享菜单中点击 **「添加到主屏幕」** |
| 3 | 确认名称后点击 **「添加」** |

#### 4.3.2 Chrome / Edge（iPad）

| 步骤 | 操作说明 |
|------|----------|
| 1 | 点击 **「⋯」** 或 **「分享」** |
| 2 | 在弹出项中选择 **「添加到主屏幕」** 或通过系统分享找到该选项 |
| 3 | 点击 **「添加」** 完成 |

---

### 4.4 前端展示策略建议

| 检测到环境 | 建议展示的引导 |
|------------|----------------|
| Android + Chrome/Edge（且可调起安装） | 优先调起系统弹窗；失败时展示 **4.1.1** |
| Android + Samsung Internet | 展示 **4.1.2** |
| Android + Firefox | 展示 **4.1.3** |
| Android + 其它 | 展示 **4.1.5** 通用兜底 |
| iOS iPhone + Safari | 展示 **4.2.1** |
| iOS iPhone + Chrome/Edge/Firefox | 展示 **4.2.2 / 4.2.3**（与 4.2.1 步骤一致时可合并文案） |
| iPad + Safari | 展示 **4.3.1** |
| iPad + Chrome/Edge | 展示 **4.3.2** |

若无法精确识别浏览器，可仅区分 **Android** / **iOS（含 iPad）**，分别展示 4.1.5 与 4.2.1，保证至少有一种可操作路径。

## 五、非功能需求与约束

### 5.1 技术约束

| 项目 | 说明 |
|------|------|
| **HTTPS** | 生产环境须使用 HTTPS，否则 Android PWA 安装条件无法满足 |
| **Manifest** | 需提供合规的 Web App Manifest（name、short_name、start_url、display、icons 等） |
| **Service Worker** | Android Chrome 等要求注册 Service Worker 才可能触发 `beforeinstallprompt` |
| **兼容性** | 不依赖未广泛支持的 API，降级为「仅展示当前平台通用引导」 |

### 5.2 体验要求

| 项目 | 说明 |
|------|------|
| **弹窗时机** | 延迟 0.5–1 秒再出现，避免首屏被遮挡 |
| **关闭方式** | 支持「暂不」按钮与点击遮罩关闭 |
| **引导可读性** | 步骤简短、可配图，移动端字体与点击区域适中 |
| **不重复打扰** | 已添加用户（standalone）不弹窗；已选「暂不」可记录并限制再次弹出策略 |

### 5.3 数据与隐私

- 仅使用本地存储（如 localStorage）记录「是否已询问」等状态，不上报用户选择内容至服务端（若需统计可单独定义合规方案）
- 不收集设备唯一标识用于强制弹窗

## 六、验收标准（摘要）

1. 在 Android 手机/平板上，使用 Chrome 打开满足 PWA 条件的页面时，用户点击「同意」能调起系统「添加到主屏幕/安装」弹窗，或展示对应浏览器引导。
2. 在 iPhone/iPad 上，用户点击「同意」后能看到与当前浏览器一致的「添加到主屏幕」步骤说明（含 Safari / Chrome 等）。
3. 从主屏幕图标进入（standalone）时不再出现征询弹窗。
4. 用户选择「暂不」后，按既定策略不再自动弹出（或仅在策略允许时再次提示）。
5. 各浏览器引导文案与第四节表格一致，且能根据设备/浏览器类型正确切换。
