# PRD Refiner

## Instructions

当用户已经有一份「需求文档 / PRD / 要点草稿」，并要求「优化、补全、重构」时，按照下面步骤执行：

1. **识别当前状态**  
   - 判断当前文档是否包含：背景、目标、用户、场景、需求列表、验收标准等要素。  
   - 标记明显缺失或混乱的部分（例如：目标和方案混在一起、没有验收标准）。

2. **重构结构但保留原意**  
   - 以 `prd-writer` 中的标准章节为参考，对文档进行结构化重组：  
     - 背景与目标  
     - 用户画像与使用场景  
     - 用户故事 / 使用流程  
     - 功能需求清单（Must / Should / Could）  
     - 交互与文案要点  
     - 埋点与数据指标（如适用）  
     - 非功能需求  
     - 验收标准  
   - 尽量保留用户原有表述和决策，仅在不清晰处进行润色或补充。

3. **补全缺失信息（合理推断）**  
   - 在用户没有写明但根据常识、上下文可以推断的地方，进行适度补全，并用简洁方式表达。  
   - 对于真正不确定、存在多种方案的地方，集中放入「开放问题 / 风险与依赖」小节。

4. **提高可执行性**  
   - 为关键需求增加：验收标准、简单用例或边界条件。  
   - 如果后续会做 Demo 或交给研发，实现相关的部分要足够具体（如字段、状态、简单交互逻辑）。

5. **输出对比视图（可选）**  
   - 当用户希望看到修改点时，可提供：  
     - 「优化前的主要问题总结」  
     - 「优化后的结构化 PRD」  
   - 不需要逐句 diff，只要让用户清楚：结构如何变好、哪些内容是新增或澄清的。

## Examples

- 示例 1：混乱的长文重构  
  - 输入：一大段没有结构的产品想法 + 各种补充说明  
  - 行为：先简短总结当前问题，再输出一份结构清晰的 PRD，并在结尾列出「本次优化重点」。

- 示例 2：不完整 PRD 补全  
  - 输入：只有背景、目标和几个功能点的简版 PRD  
  - 行为：按标准章节补全用户故事、使用流程、验收标准和风险与依赖。
