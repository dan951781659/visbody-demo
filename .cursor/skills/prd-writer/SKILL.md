# PRD Writer

## Instructions

When the user提出一个产品想法、问题或需求，希望「生成需求文档 / PRD」时，按照下面步骤执行：

1. **澄清输入信息（如有需要）**  
   - 如果用户只给了一两句话，先尽量从上下文推断；只有在严重缺信息时再简要追问。  
   - 不要求用户使用模板，只要能表达清楚业务目标即可。

2. **统一输出格式（结构化 PRD 模板）**  
   生成的 PRD 默认使用如下章节结构（可按场景增删，但尽量保持顺序）：

   - 背景与目标  
   - 用户画像与使用场景  
   - 用户故事 / 使用流程  
   - 功能需求清单（区分 Must / Should / Could）  
   - 交互与文案要点  
   - 埋点与数据指标（如适用）  
   - 非功能需求（性能、兼容性、安全、依赖等）  
   - 验收标准

3. **语言风格与粒度**  
   - 使用简体中文，面向产品、设计、研发都能理解的表述。  
   - 每个小节使用列表 + 小标题，避免大段长文。  
   - 用户故事优先用「作为一个…，我希望…，以便…」的格式。  

4. **显式标记风险与开放问题**  
   - 如果从上下文发现有不确定点，增加「开放问题 / 风险与依赖」小节，以列表形式列出。  
   - 不强迫用户当场决定，重点是帮他看见风险。

5. **适配后续步骤**  
   - 如果用户后续提到要「生成 HTML Demo」或「交给前端实现」，在 PRD 中适当补充界面结构与交互说明，方便后续技能（如 html-demo-generator）使用。

## Examples

- 示例 1：从一句话想法生成完整 PRD  
  - 用户输入：  
    - 「做一个简单的内部 OKR 管理工具，方便团队每周同步进度。」  
  - 预期行为：  
    - 自动补全：背景、目标、核心用户、典型使用场景、功能列表、验收标准等，输出完整 PRD。

- 示例 2：补全零散笔记  
  - 用户输入：  
    - 多条要点式笔记（问题、想法、竞品信息、用户吐槽等）  
  - 预期行为：  
    - 归类整理成结构化 PRD，各章节内容用用户提供的信息为主，必要时合理推断补充。
