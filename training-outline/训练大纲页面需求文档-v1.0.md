# 训练大纲页面需求文档（v1.1）

## 1. 文档目的

本文件用于定义 `training-outline` 页面所需的数据规范，重点说明：

- 页面字段值来源
- 字段生成与计算规则
- 每个数据如何获取
- 若由 AI 输出页面内容，推荐的实现方案

说明：当前 `index.html` / `mobile-dark.html` 为 Demo 形态，包含部分静态示例文案。本文给出线上化时的标准数据化方案。  
本次 v1.1 重点同步了最新 demo 行为：训练动作媒体区可切换、有图模式全量兜底、无图模式媒体区完全隐藏并调整卡片布局。

---

## 2. 页面数据总览

页面由 6 个数据模块组成：

1. 顶部摘要（hero）
2. 关键目标（goals）
3. 阶段路线（phases）
4. 执行方案 - 训练计划（train）
5. 执行方案 - 饮食计划（diet）
6. 进度复盘（review）

建议统一后端返回 `TrainingOutlinePayload`，前端只渲染，不做业务决策。

### 2.1 字段来源总原则（结合 wellnesshub-project-background）

> 以下规则用于约束“字段从哪来、在哪端算、如何降级”。

1. **端侧定位（平台角色）**
   - 本页面为 **后台（教练/顾问）主视角页面**，展示对象是用户训练方案。
   - 数据来源是 **跨端聚合**：设备测量结果（设备侧）+ 计划编排（后台）+ 展示渲染（Web 页）。

2. **字段来源优先级（统一）**
   - `L1`：计划/测量服务实时数据（可追溯主数据）
   - `L2`：规则引擎计算结果（阶段、热量、提示、标签映射）
   - `L3`：内容服务与模板（动作说明、文案、素材）
   - `L4`：前端演示态兜底（仅 Demo 可用，线上需可控开关）

3. **权限与版本约束（必须在接口层显式返回）**
   - 角色权限：教练/店长/运营可见范围不同，接口需按 `orgId/storeId/roleScope` 裁剪。
   - 版本能力：Basic/Pro 对可用模块与展示深度不同，接口需返回 `plan_capability`。
   - 设备/报告差异：体成分、体态、围度来源设备不同，接口需返回 `measurement_source` 与时间戳。

4. **事实与假设标识**
   - 已由当前 demo 与代码验证的行为标记为“**已确认**”。
   - 依赖平台后端能力、但当前仓库未内置源文档校验的内容标记为“**待后端确认**”。

## 3.1 顶部摘要（hero）

### 3.1.1 固定字段

| 页面字段 | 前端位置 | 来源 | 获取方式 | 规则 |
|---|---|---|---|---|
| 页面标题 | `hero-title` | 文案模板库 | 配置中心/静态模板 | 固定文案 |
| 副标题 | `hero-sub` | 文案模板库 | 配置中心/静态模板 | 固定文案，可按产品线差异化 |
| 周期徽标 | `hero-badge` | 计划引擎 | 计划服务接口 | `起始周/总周数/每周训练次数` |
| 预计可见变化 | `metric-value` | 规则引擎 | 目标服务 | 依据用户分层给出周数文案 |

### 3.1.2 动态指标卡（hero-metrics）

页面展示 **3 个指标卡**，不固定为"体脂率/腰围/体态"，而是从用户已完成的测量项目中，取严重度最高的 3 个异常指标。

#### 3.1.2.1 各测量类型候选异常指标池

| 测量类型 | 候选异常指标 | 指标标签 | 目标值格式 | 判定标准 |
|---|---|---|---|---|
| 体成分 | 体脂率偏高 | 体脂率目标 | `32% -> 26%` | 超出性别年龄标准上限 |
| 体成分 | BMI 超重/肥胖 | BMI 目标 | `27.2 -> 23.5` | BMI >= 24（超重）或 >= 28（肥胖） |
| 体成分 | 内脏脂肪偏高 | 内脏脂肪目标 | `12级 -> 8级` | 等级 >= 10 |
| 体成分 | 肌肉量不足 | 肌肉量目标 | `38kg -> 42kg` | 低于性别年龄标准下限 |
| 体成分 | 基础代谢偏低 | 基础代谢目标 | `1180 -> 1320kcal` | 低于预测值 85% |
| 体围 | 腰围超标 | 腰围目标 | `90cm -> 84cm` | 男 >= 90cm / 女 >= 80cm |
| 体围 | 腰臀比超标 | 腰臀比目标 | `0.92 -> 0.85` | 男 >= 0.9 / 女 >= 0.85 |
| 体围 | 大腿围偏大 | 大腿围目标 | `62cm -> 57cm` | 超出同身高性别 P75 |
| 体态 | 圆肩 | 体态重点 | `圆肩改善` | 体态评分中圆肩项异常 |
| 体态 | 驼背 | 体态重点 | `驼背改善` | 体态评分中驼背项异常 |
| 体态 | 骨盆前倾 | 体态重点 | `骨盆前倾改善` | 体态评分中骨盆前倾项异常 |
| 体态 | 高低肩 | 体态重点 | `高低肩改善` | 左右肩高差 >= 2cm |
| 体态 | 头前倾 | 体态重点 | `头前倾改善` | 耳垂-肩峰偏移 >= 3cm |
| 健康档案 | 久坐风险 | 活动建议 | `日均久坐 -> 每小时起身` | 日均久坐 >= 8h |
| 健康档案 | 睡眠不足 | 睡眠建议 | `5.5h -> 7h+` | 日均睡眠 < 6.5h |

#### 3.1.2.2 严重度评分规则

统一打分公式，使跨类型可比：

| 类型 | 评分方式 | 示例 |
|---|---|---|
| 数值类（体脂率/BMI/腰围等） | `severity = (实际值 - 标准上限) / 标准上限 * 100` | 体脂率 32%，标准上限 25%，severity = 28 |
| 等级类（内脏脂肪） | `severity = (实际等级 - 安全等级) * 10` | 等级 12，安全 9，severity = 30 |
| 布尔类（体态/健康档案） | 按预定义优先级赋固定分 | 圆肩 = 25，高低肩 = 15，久坐 = 12 |

体态类固定优先级参考（可配置）：

| 体态项 | 默认 severity |
|---|---|
| 驼背 | 30 |
| 圆肩 | 25 |
| 骨盆前倾 | 22 |
| 头前倾 | 18 |
| 高低肩 | 15 |

#### 3.1.2.3 指标选取规则

1. 从用户**已完成**的测量类型中，汇总所有候选异常指标
2. 计算每个指标的 severity 分
3. 按 severity **降序排列**，取 top 3
4. 如果异常不足 3 项，用**正向维持指标**补位（见兜底规则）

#### 3.1.2.4 不同测量组合示例

| 已完成测量 | 指标卡 1（最严重） | 指标卡 2 | 指标卡 3 |
|---|---|---|---|
| 仅体成分 | 体脂率偏高 `32% -> 26%` | 内脏脂肪偏高 `12级 -> 8级` | BMI超重 `27.2 -> 23.5` |
| 体成分 + 体围 | 腰围超标 `90cm -> 84cm` | 体脂率偏高 `32% -> 26%` | 腰臀比超标 `0.92 -> 0.85` |
| 体成分 + 体态 | 体脂率偏高 `32% -> 26%` | 圆肩 `圆肩改善` | BMI超重 `27.2 -> 23.5` |
| 体成分 + 体围 + 体态 | 腰围超标 `90cm -> 84cm` | 体脂率偏高 `32% -> 26%` | 圆肩 `圆肩改善` |
| 全部测量 | 按 severity 混排取 top 3 | - | - |
| 仅体态 | 驼背 `驼背改善` | 圆肩 `圆肩改善` | 骨盆前倾 `骨盆前倾改善` |

#### 3.1.2.5 兜底规则（异常不足 3 项）

| 情况 | 处理 |
|---|---|
| 异常 = 2 | 第 3 卡显示最接近阈值的正常指标，标签改为"维持目标"，如 `肌肉量 42kg -> 维持` |
| 异常 = 1 | 第 2/3 卡补充正向维持指标 |
| 异常 = 0 | 3 卡均显示"维持目标"类指标，侧重体能/代谢/体态维持 |
| 未做任何测量 | 不应进入此页面（前置流程拦截） |

#### 3.1.2.6 指标卡展示格式

每张指标卡统一结构：

- **标签**（`metric-label`）：指标名称，如"体脂率目标"、"腰围目标"、"体态重点"
- **数值**（`metric-value`）：
  - 数值类：`当前值 -> 目标值`
  - 体态类：`问题名称 + 改善`
  - 维持类：`当前值 -> 维持`

### 3.5.3 训练规则（当前代码）

- `getStageByWeek(week)`：周映射到阶段焦点

### 3.6.2 食谱条目字段

| 字段 | 来源 | 获取方式 | 规则 |
|---|---|---|---|
| `time` | 餐次计划 | 饮食计划服务 | HH:mm |
| 餐型标签 | `mealLabelFromTime(time)` | 前端规则 | <10 早餐，<17 午餐/加餐，否则晚餐/夜间 |
| `text` | 食谱模板/AI | 饮食计划服务/AI | 食材+份量描述 |
| `macro` | 营养估算服务 | 营养服务 | 格式 `Pxx/Cxx/Fxx` |
| 餐次热量 | `calcMealKcal(macro)` | 前端计算 | `P*4 + C*4 + F*9` |
| 餐次目标短语 | `getMealGoalText()` | 关键词规则 | 从 detail.goal 提取语义或按时段兜底 |

### 3.6.3 食谱详情弹窗字段

字段：`goal/ingredients/replace/timing/caution/steps/prepTime/outOption/avoid`  
来源：优先 `meal.detail`，缺失走默认兜底模板。

### 3.6.4 饮食计算规则（当前代码）

1) **总热量修正**

- `baseKcal = dietPlan[day].kcal`
- `weekOffset = 0`
  - 周 3-6：`-80`
  - 周 7-8：`-30`
- `adjustedKcal = baseKcal + weekOffset + manualAdjustments[week]?.kcalDelta`

2) **宏量分配**

- `protein(g) = max(100, round(adjustedKcal * 0.24 / 4))`
- `carb(g) = round(adjustedKcal * 0.50 / 4)`
- `fat(g) = round(adjustedKcal * 0.26 / 9)`

3) **饮水目标**

- 周 1-2：`2.0L`
- 周 3-8：`2.2L`

## 4. 数据获取方案（后端接口建议）

## 4.1 建议接口

1. `GET /api/user/{userId}/measurement-latest`
- 体成分、体围、体态、健康档案最新数据

2. `POST /api/training-outline/plan/generate`
- 输入：用户基础信息 + 目标 + 周期参数
- 输出：训练计划主结构（周、日、动作）

3. `POST /api/nutrition-outline/plan/generate`
- 输入：用户体重、目标、训练日历、饮食偏好、禁忌
- 输出：每日热量、宏量、食谱条目、餐次详情

4. `GET /api/content/exercise/{actionId}`
- 动作知识库详情（步骤/呼吸/常错/回退/进阶）

5. `GET /api/content/review-rules?week={n}`
- 复盘规则文案（expected/normal/retest/focus/metrics）

## 4.2 页面聚合接口（推荐）

`GET /api/training-outline/page?userId={id}&week={n}&day={d}`

一次返回页面渲染所需全部字段，前端只负责状态切换和展示。

建议新增返回元信息（用于字段来源审计）：

- `data_permissions`：角色与组织范围

## 5. AI 输出方案（如何让 AI 生成这部分内容）

## 5.1 推荐架构

采用“**规则先算 + AI 后写**”：

- 数值类（热量、宏量、周阶段）由规则引擎计算
- AI 只生成说明文案、动作提示、食谱描述、复盘文案
- AI 不直接决定医学或营养底层参数

## 5.2 AI 输入（结构化）

## 5.3 AI 输出（必须 JSON）

输出结构必须与前端字段一一对齐，建议包含：

- `hero`
- `goals`
- `phases`
- `train_plan`（对应 `trainPlan`）
- `diet_plan`（对应 `dietPlan`）
- `review_by_week`（对应 `weekMap`）

并增加 `validation_flags`，用于后置质检。

## 5.4 提示词关键约束（System Prompt）

- 禁止捏造医学结论
- 禁止输出与输入冲突的数据
- 动作和食谱文案必须“可执行”
- 单条文案长度限制（如 20~60 字）
- 统一术语：训练强度、时长、器械、注意事项

## 5.5 校验与回退

1. **结构校验**：JSON Schema 必须通过  
2. **数值校验**：热量与宏量一致性校验（4/4/9）  
3. **敏感词校验**：医疗诊断类词汇拦截  
4. **回退策略**：AI 失败时降级到模板库

## 6. 验收标准（面向研发）

- 字段完整率：100%（页面所有展示字段有来源）
- 规则可复算：热量/宏量计算可复现
- 数据可追溯：每个字段可定位到接口或规则
- AI 生成稳定：同输入多次生成结构一致
- 缺失容错：任一模块缺数据不致页面崩溃
